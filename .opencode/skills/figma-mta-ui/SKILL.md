# Figma to MTA UI

Use this workflow whenever the user sends a Figma link, selected frame, `node-id`, screenshot, mockup, or asks to recreate UI from Figma in this repository.

## Goal

Implement the design as an MTA CEF view owned by the feature resource and mounted through `fh-ui`. Match the supplied design closely while fitting the existing Forza-style interface and the project's lightweight architecture.

## Read first

- Inspect `AGENT_STATE.md` for the current architecture.
- Inspect `fh-ui/README.md`, `fh-ui/client.lua`, and `fh-ui/web/js/runtime.js` before adding a new pattern.
- Inspect the closest existing feature, especially its `client.lua`, `meta.xml`, and `ui/` files. Good references are `fh-login`, `fh-hud`, and `fh-dashboard`.
- Reuse the existing visual language when extending an established screen. Do not replace the project's style with a generic web dashboard.

## Activation Rule

Use this skill only when the current task matches this domain, project type, file type, reference material, or risk profile. Select it through skill-router or obvious task context; do not keep all skills active at once. Inspect relevant files and verify results before final response.

- Use the project MCP server named `Figma Desktop` at `http://127.0.0.1:3845/mcp` when Figma context is available.
- If the user provides a Figma link or `node-id`, inspect that exact frame or component first.
- If the user refers to the current selection, inspect the selected Figma node before coding.
- Always prefer `get_design_context` for the target node, then use `get_metadata` for structure/child IDs when the design is complex, and `get_screenshot` to visually compare the result.
- Use `get_variable_defs` when the design uses variables/tokens. Map tokens to CSS variables instead of guessing colors or spacing.
- Collect exact layout dimensions, spacing, typography, colors, opacity, borders, shadows, assets, variants, constraints, and responsive behavior before editing files.
- Export real icons/assets from Figma or use a matching open-source SVG icon only when Figma's icon cannot be exported. Prefer SVG for icons and PNG/WebP only for raster imagery. Keep assets in the owner resource under `ui/assets/`, `ui/icons/`, or its existing asset folder.
- Treat generated web code as reference only. Write project-native HTML/CSS/JS instead of pasting React, Tailwind, or framework output.

## Measurement-first workflow

Before coding, build a small measurement notes list from Figma. Do not rely on visual guessing when the MCP output contains numeric values.

For complex frames, create a compact implementation map before editing:

- target component name and owner resource;
- root frame size and scale strategy;
- major regions and DOM groups;
- repeated components that should be generated by JS versus static HTML;
- Lua data payload keys needed by each region;
- assets/icons that must be exported or recreated;
- animation states and approximate durations.

Capture and preserve:

- **Frame**: width, height, background fill, clipping, safe area, absolute position only for reference.
- **Layout mode**: auto-layout direction, wrap, padding top/right/bottom/left, item spacing/gap, alignment, distribution, hug/fill/fixed sizing.
- **Element boxes**: x/y within parent, width, height, min/max where inferable, border radius per corner, stroke width/alignment.
- **Typography**: font family, font size, line height, weight, letter spacing, paragraph spacing, text transform/case, alignment, color, opacity.
- **Colors**: fill/stroke/shadow colors with alpha; prefer variables from `get_variable_defs` when available.
- **Effects**: box shadows, inner shadows, blur, layer opacity, blend mode, gradient stops and angles.
- **Assets/icons**: node name, SVG/raster type, dimensions, intended color behavior, export path.

When a value is missing from generated code, inspect the node metadata or child node instead of inventing a number. If the value still cannot be read, state the uncertainty and choose the closest visible value.

## Spacing and sizing rules

- Reproduce Figma auto-layout with CSS Grid/Flex using the exact `gap` and padding values from Figma.
- Do not use arbitrary `margin: 20px`, `gap: 16px`, card widths, or font sizes just because they look common. Use measured values first.
- If Figma uses absolute positioning, convert to relative CSS only after preserving the measured box sizes and offsets.
- Keep card/window spacing consistent with Figma: gutters, columns, row gaps, internal padding, icon-label gaps, and header/body separation must be measured separately.
- Use `box-sizing: border-box` so CSS width/height matches Figma boxes with strokes.
- Use `clamp()`/`vw`/`vh` only after the desktop reference is accurate. Responsive scaling must not destroy the original proportions.
- If a design is made for a fixed game resolution, implement a scaled root wrapper rather than reflowing every child differently.

## Typography fidelity

- Use the exact Figma font family when the repo already has it or when a local font asset can be added safely.
- If the exact font is unavailable, pick a close local fallback and say it is a fallback.
- Preserve font size, line height, weight, letter spacing, uppercase/lowercase, alignment, and opacity.
- Do not collapse multiple text styles into one generic body style.
- Keep numeric/stat text styles separate from labels and headings.

## SVG icons and Figma assets

- Load/use the `svg-icon-ui` skill whenever implementing icons from Figma.
- Do not recreate Figma icons with CSS pseudo-elements, numbers, emoji, or text glyphs.
- If the Figma node is an icon/vector, export or save it as a local `.svg` whenever possible.
- Sanitize SVGs: remove scripts, comments, metadata, hidden editor junk, and hard-coded colors unless the icon has intentional brand colors.
- For recolorable UI icons, use `stroke="currentColor"` or `fill="currentColor"` and control color from CSS states.
- Store icons in `fh-RESOURCE/ui/icons/*.svg` and add each one to `meta.xml`.
- Reference SVGs through CSS masks for single-color icons, or inline SVG for multicolor/animated icons.
- Preserve icon size from Figma, including the icon box and visual path alignment. Do not stretch icons to fill a button if Figma used optical padding.
- Match icon library/style across a component. Do not mix outline, filled, rounded, and sharp sets unless Figma does.

Recommended local mask pattern:

```css
.component__icon {
    width: 20px;
    height: 20px;
    flex: 0 0 20px;
    background: currentColor;
    -webkit-mask: var(--icon-url) center / contain no-repeat;
    mask: var(--icon-url) center / contain no-repeat;
}

.component__icon--settings { --icon-url: url("icons/settings.svg"); }
.component__button { color: var(--icon-muted); }
.component__button.is-active { color: var(--accent); }
```

## Color and effect fidelity

- Use CSS variables for Figma variables/tokens and repeated colors.
- Preserve alpha exactly when possible: `rgba(...)`, 8-digit hex, or opacity on a wrapper when that matches Figma.
- Recreate shadows with exact x/y/blur/spread/color from Figma. Do not replace all shadows with a generic glow.
- Recreate gradients with the same stops and direction. If the MCP code uses absolute gradient handles, approximate angle and document if uncertain.
- Keep border radii and stroke widths exact. Figma 1px stroke usually maps to `border: 1px solid ...` with `box-sizing: border-box`.

## Project UI Ownership Map

Use the current project architecture when deciding where a Figma/UI implementation belongs:

- Shared runtime/bootstrap only -> `fh-ui`.
- Login screen and cinematic login data -> `fh-login/ui`, not `fh-ui`.
- Speedometer/driving HUD -> `fh-hud/ui`.
- Dashboard/pause/map/stats/accolades -> `fh-dashboard/ui`.
- Admin workspace -> `fh-admin/ui`.
- Radio overlay -> `fh-radio/ui`; keep left-above-radar placement and `[`/`]` station control.
- Points/skill chain widget -> `fh-points/ui`.
- Level-up animation -> `fh-progression/ui`.
- Discovery/weather/photo/race views -> their owner resources: `fh-discovery`, `fh-weather`, `fh-photo-mode`, `fh-races`, `fh-race-finish`, `wheelspin-test`.
- Future shared primitives such as notifications, keybar, modal dialogs, settings, audio, or route provider should become separate focused resources only when explicitly requested or clearly reusable.

Project-specific UI rules:

- Do not put one-off feature HTML/CSS into `fh-ui` just because it is CEF.
- Use `data-ui-text`, `data-ui-visible`, `data-ui-src`, `data-ui-class`, `data-ui-style`, and `data-ui-var` before adding custom JS.
- Owner resources must list all UI files/assets in their own `meta.xml`.
- After UI changes, the usual validation path is owner resource restart plus `/fhuireload` or the project reload overlay when relevant.
- Keep Forza/Horizon-style density, cinematic overlays, glass/dark gradients, map treatment, and key hints consistent with existing components.

## Architecture rules

- Keep one shared fullscreen CEF browser in `fh-ui`. Do not create another browser for ordinary UI.
- Load `mta-resource-development` too when the Figma implementation touches Lua, `meta.xml`, commands, keybinds, events, exports, database, or autostart.
- Put feature markup and styles in the owner resource, usually `fh-<feature>/ui/<view>.html` and `fh-<feature>/ui/<view>.css`.
- Add every CEF file and asset to the owner resource `meta.xml` with `<file src="..." />`.
- Register before first show:

```lua
exports['fh-ui']:registerComponent('view-name', 'fh-feature', 'ui/view.html', 'ui/view.css', 'ui/view.js')
exports['fh-ui']:showComponent('view-name', data)
```

- Omit the JS path if HTML/CSS plus runtime bindings are enough.
- Hide temporary views with `hideComponent`; the runtime unmounts them from DOM after the exit animation.
- Patch live values with `patchComponentData` instead of remounting or repeatedly sending full state.
- Enable input with `setInputMode(true)` only for interactive panels and restore it on close.
- Unregister owner views during resource stop when appropriate:

```lua
exports['fh-ui']:unregisterResourceComponents(getResourceName(resource))
```

## Prefer runtime bindings

Use the built-in attributes before adding custom JS:

| Attribute | Purpose |
| --- | --- |
| `data-ui-text="money"` | Set text from Lua data |
| `data-ui-default="..."` | Fallback text |
| `data-ui-visible="isVisible"` | Toggle `hidden` |
| `data-ui-src="image"` | Set image `src` |
| `data-ui-option="garage"` | Apply `is-selected` from `selection` |
| `data-ui-class="hasError:is-error"` | Toggle class from boolean data |
| `data-ui-style="width:progress:%"` | Set inline style from data |
| `data-ui-var="--progress:progress:%"` | Set CSS variable from data |

Add `ui/<view>.js` only when the screen needs DOM generation, click handlers, MTA events, or non-trivial client-side logic. Keep Lua responsible for game state and JS responsible for presentation behavior.

## UI Engineering Standard

For large interactive UI, design the component contract before writing markup:

- Define component name, owner resource, visibility lifecycle, input mode behavior, close/toggle commands, and data payload shape.
- Split static shell, repeated list/card items, state classes, and animation phases clearly.
- Keep one root class per component and scope every selector under it.
- Use CSS variables for repeated colors, spacing, radii, shadows, animation durations, z-index, and scale.
- Use JS only for data-driven DOM generation, click/key interaction, state sequencing, and animation replay. Do not move game rules into JS.
- For large lists, render only needed DOM and update changed values rather than remounting the whole component every tick.
- For interactive panels, support mouse and keyboard close paths and ensure focus/cursor/input mode cannot remain stuck after hide/resource stop.
- Use empty/loading/error states for data that may arrive late from Lua/server.
- Keep text overflow rules explicit: ellipsis, wrapping, numeric alignment, unit formatting, and minimum readable sizes.

## Design translation

- Match the Figma frame at its intended desktop reference size, then make it safe for common MTA resolutions with `%`, `vw`, `vh`, `min()`, `max()`, or `clamp()` where useful.
- Use scoped BEM-like classes such as `.fh-garage__tile`; avoid broad selectors that leak into other mounted components.
- Keep the root background transparent when the game world or Lua cinematic should remain visible.
- Recreate gradients, overlays, shapes, shadows, and a small number of meaningful enter/exit transitions from the design.
- Preserve existing project conventions for accent colors, typography, motion, map treatment, and key hints when the frame belongs to an existing flow.
- Do not use React, Vue, package installs, a build step, external CDNs, or web fonts that may be unavailable inside MTA CEF unless the repository already provides them.

## Figma comparison checklist

Before finalizing, compare against the Figma screenshot/reference at the same viewport size:

- Root frame size and alignment match.
- Card/window width, height, padding, gap, and border radius match.
- Text font size, weight, line height, and color match.
- Icons are actual local SVGs, correctly sized and colored.
- Backgrounds, gradients, borders, shadows, opacity, and blur match.
- Lists/grids use the same row height, column gaps, and internal spacing.
- Buttons/inputs match height, padding, radius, label position, and state colors.
- No element is shifted because of default browser margins, unaccounted border-box differences, or missing Figma auto-layout padding.

If the implementation visibly differs, fix the measured CSS values instead of explaining the difference away.

## MTA boundary

- HTML/CSS/JS implements visual presentation only.
- Lua handles game state, vehicles, camera, timers, keybinds, server events, and data preparation.
- Use `triggerEvent` or the repository's existing browser event pattern only when CEF must notify Lua about direct pointer interaction.
- Keep high-frequency patches small and throttle them similarly to existing HUD code.
- Never let CEF decide economy, rewards, ownership, account ids, race completion, admin permissions, or destructive actions. Send user intent to Lua/server and wait for authoritative state.
- For interactive Figma panels, define focus/cursor/input lifecycle: open, close, `Esc`, resource stop, failed dependency, and external hide.

## Advanced Fidelity Rules

- Preserve Figma hierarchy before decoration. If time is limited, correct size, position, typography, and spacing before adding extra effects.
- Match repeated component rhythm exactly: card heights, row gaps, list item padding, icon boxes, divider offsets, and selected-state thickness.
- Distinguish content opacity from panel opacity. If Figma uses translucent panel fill, do not apply opacity to the whole container and fade text/icons unintentionally.
- Do not replace measured auto-layout with many unrelated absolute offsets unless the reference is genuinely absolute-positioned.
- For lists/grids with dynamic data, build the first static row/card to match Figma, then generate repeated items from JS using the same class structure.
- Preserve empty/negative space. Do not fill blank Figma areas with invented controls or stats.
- If the Figma frame includes prototype variants or components, inspect the relevant selected/hover/disabled state instead of creating one generic state.

## UI Regression Checklist

Before finalizing complex CEF UI, check:

- no unscoped selectors can affect other mounted components;
- no remote/CDN/font dependency was introduced;
- all CSS asset URLs are relative to the CSS file location and listed in `meta.xml`;
- runtime binding keys match Lua payload keys exactly;
- temporary components unmount through `hideComponent` and persistent widgets avoid remount storms;
- interactive panels release cursor/input mode on close, `Esc`, resource stop, and dependency failure;
- z-index/layering does not cover chat, radar, speedometer, admin panel, dashboard, or reload overlay unexpectedly;
- UI remains readable at 16:9, smaller windows, and ultrawide enough for normal MTA use;
- animations have reduced risk of stuck state by using classes/timeouts consistently.

## Verification

- Confirm new HTML, CSS, JS, and assets are listed in `meta.xml`.
- Confirm `meta.xml` script order still loads config/helpers before entrypoints and does not add dependency `<include>` entries.
- Confirm local SVG icons/assets exist at the referenced paths and are included in `meta.xml`.
- Confirm the component is registered before it is shown.
- Confirm temporary views close through `hideComponent` and interactive views release input mode.
- Run basic syntax/path checks when possible: JS syntax, `meta.xml` file references, and missing asset paths.
- Use `/fhuireload` or `F6` for CEF iteration and `/browserdebug` for DevTools when validating in MTA.
- If MTA cannot be launched from the current environment, state that visual verification still needs an in-game pass.

## Reporting requirements

When implementing from Figma, report briefly:

- Figma node inspected and whether `get_design_context`, `get_metadata`, `get_screenshot`, and `get_variable_defs` were used.
- Reference frame size and important measured values used: main gaps, padding, font sizes, and card sizes.
- Assets/icons exported or substituted, with local paths.
- Files changed and verification performed.
- Any Figma values that could not be read exactly and how they were approximated.
